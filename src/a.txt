================================================================================
AASAMEDCHEM: B2B MULTI-VENDOR MARKETPLACE SYSTEM WORKFLOW
================================================================================

This document explains the high-level architecture, user roles, database model,
unit conversions, and platform commission workflows of the AasaMedChem inventory
and order management system.

--------------------------------------------------------------------------------
1. SYSTEM ARCHITECTURE & ROLES
--------------------------------------------------------------------------------
The system is built on a full-stack Next.js architecture connected directly to a
Neon PostgreSQL database. Authentication is handled using secure HttpOnly JWT 
cookies. Next.js edge-compatible middleware protects role boundaries.

The platform is designed around three distinct roles:
1. User (Buyer): A customer (e.g. pharmacy or distributor) who browses products,
   places order quotations using flexible unit selections, and requests custom
   medicines if they are out of stock.
2. Seller (Supplier): A supplier who lists their own chemical products, defines
   their preferred base unit and base price, maintains inventory stock, and logs
   received orders.
3. Admin (Marketplace Coordinator): An administrator who oversees transactions,
   approves or rejects order quotations, coordinates custom medicine requests, and
   collects platform commissions.

--------------------------------------------------------------------------------
2. DATABASE MODEL & STRUCTURE (Neon PostgreSQL)
--------------------------------------------------------------------------------
The database is structured to securely isolate B2B actors and maintain financial
auditing:
* users: Stores hashed credentials and role classifications ('admin', 'seller', 'user').
* products: Houses catalog listings. Each product is linked to a supplier via
  `seller_id`. Prices and stock counts use `NUMERIC(20, 6)` for high precision.
* orders: Records order transactions. Keeps track of the total invoice and approval status.
* order_items: Audits order details. Stores the quantity, ordering unit, base price
  at checkout, and the conversion factor applied.
* medicine_requests: Pipeline for unavailable medicines, including customer phone,
  requested drug details, and status.

--------------------------------------------------------------------------------
3. INVENTORY, UNIT CONVERSIONS & CHECKOUT CHANNELS
--------------------------------------------------------------------------------
Medicines are listed in base units (like kg, g, mg, L, mL, or item). Buyers are
allowed to order in compatible units of the same dimension (e.g., ordering 'mg'
when a product is stored in 'kg').

There are two distinct checkout pathways:
1. DIRECT BUY (Instant Purchase):
   - Deducts stock immediately from the seller's inventory in the database.
   - Blocks the transaction immediately if the requested quantity exceeds stock.
   - Sets order status directly to 'completed'.
2. REQUEST QUOTATION:
   - Does NOT deduct stock immediately (allowing placement even if quantity exceeds current stock).
   - Show a warning in the cart if quantity exceeds stock, but lets the user submit the quote.
   - Sets order status to 'pending'. Stock is only deducted when the Admin approves it.

Here is how a checkout transaction is processed in the code:
1. Selection: The Buyer adds a product to their cart.
2. Conversion: When quantity and unit are selected, Big.js computes:
   - Converted Qty = Ordered Qty * Conversion Factor (e.g., 1 mg = 0.000001 kg)
   - Line Price = Converted Qty * Base Price
3. Database Transaction & Locking:
   - The API opens a transaction and locks the product rows using `FOR UPDATE` to prevent double-selling.
   - For Direct Buy: checks stock, rejects if insufficient, deducts stock, and writes orders.
   - For Quotation: skips stock check, creates order as 'pending' with 0 stock deduction.
4. Admin Status Changes:
   - If the Admin approves a pending quote, the system checks stock and deducts it.
   - If rejected, no stock deduction occurs. If a direct buy order is rejected, stock is restored.

--------------------------------------------------------------------------------
4. PLATFORM COMMISSIONS & GMV TRACKING
--------------------------------------------------------------------------------
AasaMedChem charges a 5.0% platform fee on all B2B transactions:
* Gross Merchandise Value (GMV): The total value of an order invoiced to the Buyer.
* Platform Commission: 5% of the GMV, which goes to the Admin.
* Seller Payout: 95% of the GMV, which goes to the Supplier.

How the views adapt:
* Admin View: The Admin sees gross transaction volume cards (GMV), booked commissions
  (5% of all orders), cleared GMV (approved/completed orders), and cleared platform
  revenue. Selecting an order shows the complete payout audit.
* Seller View: The Seller sees orders placed on their products. Instead of the full
  invoice total, the details show the gross transaction value, the 5% platform fee
  deducted, and their Net Supplier Payout (95%).

--------------------------------------------------------------------------------
5. CUSTOM MEDICINE REQUEST PIPELINE
--------------------------------------------------------------------------------
If a User searches for a drug and it is not in the system catalog:
1. The User clicks "Request Custom Medicine" from the search fallback card.
2. They input their Display Name, Phone Number, Medicine Name, Required Qty/Unit,
   and Delivery Address.
3. This is saved to `medicine_requests` and appears on the Admin dashboard.
4. The Admin contacts Sellers in the backend to source the drug.
5. Catalog Integration: Once a supplier is found, the Admin clicks "Add to Catalog"
   on the request card. This redirects to the products list and automatically opens
   the product creation form pre-filled with the name, requested unit, and details.
6. The Admin selects a Seller, configures the base price, and saves it. It is now
   instantly searchable in the marketplace!

================================================================================
7. GOOGLE FORM SHORT-ANSWER CHEAT SHEET
================================================================================

Q1: In which units do you store product quantities in the database?
A1: Product quantities are stored strictly in the configured product's "base_unit" (e.g. mg, g, kg, mL, L, or item). Prices are stored as rate-per-base-unit.

Q2: Which data type(s) did you use in the database for storing quantities and prices?
A2: Exact PostgreSQL NUMERIC data types: NUMERIC(20, 6) for quantities, prices, and line items, and NUMERIC(20, 10) for exact conversion factors.

Q3: Briefly describe the conversion factors you use between grams and kilograms, and between milliliters and liters.
A3: We use base conversion ratios:
- Weight: 1 kg = 1000 g (Factor: 1000.00); 1 g = 0.001 kg (Factor: 0.0010); 1 g = 1000 mg (Factor: 1000.00).
- Volume: 1 L = 1000 mL (Factor: 1000.00); 1 mL = 0.001 L (Factor: 0.0010).
Conversions across physical dimensions are blocked.

Q4: How do you ensure that quantities are stored consistently while still displaying them in different units in the UI?
A4: We enforce a strict "Base Unit Storage" rule. The database stores all stock quantities and prices exclusively in terms of the product's configured base_unit. When displaying a product in the UI, we retrieve this base quantity and price, and if the user selects a compatible ordering unit, we compute the converted display values dynamically using exact math (Big.js). To maintain auditability, when an order is placed, the API records both the user's selected unit/quantity and the exact conversion_factor_used, enabling the UI to reconstruct the exact calculations of the order at checkout.

Q5: Mention one or two edge cases you considered for unit conversion or pricing (for example, rounding, very small or very large values).
A5: We considered two primary edge cases:
1. Micro-precision scaling: Chemical products can require transactions in micrograms (mg on a kg base). Floating-point calculation is unsafe here. We use Big.js and PostgreSQL NUMERIC data types to preserve exact precision down to 6 decimals for storage and 10 decimals for conversion factors.
2. Concurrent stock race conditions: When multiple buyers checkout simultaneously, a race condition can cause double-selling. We lock rows at checkout using SELECT FOR UPDATE to serialize stock evaluations and deductions.

Q6: If you wanted to add support for a new unit or dimension (for example, pounds or gallons), how would you extend your current design?
A6: To extend the design, we would:
1. Add the new units (e.g. lbs, gal) to the database CHECK constraints on the products, order_items, and medicine_requests tables.
2. Update conversions.ts:
   - If adding a unit to an existing dimension (e.g., lbs to Weight), add its conversion factor relative to the base (e.g., 1 kg = 2.20462 lbs).
   - If adding a new dimension (e.g., Imperial Volume with gal), define a new dimension mapping and list compatible units and their factors relative to a base (e.g., gal and qt with base gal).
3. The rest of the checkout, rounding, and audit code is dimension-agnostic and will automatically support it.
